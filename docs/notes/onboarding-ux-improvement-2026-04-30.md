# kube-slint 외부 온보딩 UX 개선 계획

- 작성일: 2026-04-30
- 상태: 검토 중 (Draft)
- 목적: 외부 팀이 kube-slint를 도입할 때 겪는 마찰을 제거하고, 문서나 내부 컨텍스트 없이도 첫 PASS를 받을 수 있는 경험을 설계함

---

## 1. 문제 정의

kube-slint는 오퍼레이터 개발 단계에서 SLI를 검증하는 shift-left 도구다. 설계 철학과 구현 품질은 검증되었지만, **외부 팀이 혼자 도입할 때 막히는 지점이 너무 많고, 너무 일찍 나온다.**

현재 내부에서는 JUMI, AH 등 다수 프로젝트에서 사용 중이다. 그러나 이들은 팀 내 컨텍스트(서비스 이름, RBAC 패턴, 임계값 기준)를 이미 알고 있어서 진입 비용이 낮다. 외부 팀에게는 그 컨텍스트가 없다.

---

## 2. 현재 온보딩 경로 분석

외부 팀이 `go get github.com/HeaInSeo/kube-slint` 부터 첫 PASS까지 가는 실제 경로:

```
Step 1  go get github.com/HeaInSeo/kube-slint
        → 쉬움

Step 2  SessionConfig 작성
        → MetricsServiceName을 어떻게 아나?
          kubectl get svc를 직접 뒤져야 한다.
          이름 규칙을 모르면 막힘.

Step 3  RBAC 설정 (pods: create)
        → 문서에 한 줄 주석으로만 언급됨.
          설정 안 하면 curl-pod 생성 실패.
          에러 메시지: "pods is forbidden"
          원인을 모르면 디버깅에 수십 분 소요.

Step 4  CurlImage 결정
        → 내부 레지스트리가 있는 팀은 이미지 경로를 직접 알아야 함.
          공개 이미지 기본값이 없음.
          에어갭(air-gap) 환경에서는 추가 작업 필요.

Step 5  E2E 실행 → sli-summary.json 생성
        → 클러스터가 없으면 전혀 진행 불가.
          로컬에서 미리 검증할 방법이 없음.

Step 6  policy.yaml 작성
        → 임계값을 뭘로 써야 하는지 아무도 모름.
          새로 합류한 팀은 reconcile_total_delta >= 1이 맞는지
          workqueue_depth_end <= 5가 적절한지 판단 근거가 없음.

Step 7  slint-gate 실행
        → 비교적 쉬움

Step 8  첫 PASS 확인
        → 여기까지 도달하는 팀이 드뭄
```

### 마찰 강도 요약

| 단계 | 마찰 강도 | 이유 |
|---|---|---|
| Step 2 MetricsServiceName | ★★★ | 클러스터 직접 조회 필요 |
| Step 3 RBAC | ★★★★★ | 에러 메시지로 원인 파악 불가 |
| Step 4 CurlImage | ★★★ | 기본값 없음, 레지스트리 의존 |
| Step 5 클러스터 의존 | ★★★★ | 로컬 검증 불가 |
| Step 6 policy.yaml | ★★★★ | 임계값 판단 근거 없음 |

## 3. 개선 목표

> **외부 팀이 README만 보고, 문서 없이, `slint-gate init` 하나로 첫 PASS를 받을 수 있어야 한다.**

이것이 달성되면 나머지 커스터마이징(임계값 조정, 회귀 감지 설정 등)은 사용자가 스스로 탐색한다.

---

## 4. 개선 방향 (우선순위 순)

### 4-1. `slint-gate init` 명령어 [최우선]

**목적**: "policy.yaml을 뭘로 써야 하나" 문제를 제거한다.

**동작**:
```sh
slint-gate init --namespace my-operator-system
```

실행 시:
1. 지정 namespace에서 `/metrics` 엔드포인트를 가진 서비스를 자동 탐색
2. 실제 메트릭을 스크랩해서 사용 가능한 메트릭 목록 파악
3. kubebuilder 계열 오퍼레이터 표준 메트릭(`controller_runtime_*`, `workqueue_*`)을 인식
4. 현재 메트릭 값을 기반으로 **현실적인 초기 임계값**이 채워진 `.slint/policy.yaml` 생성
5. `SessionConfig` 코드 스니펫을 stdout에 출력

**생성 예시**:
```yaml
# .slint/policy.yaml
# Generated by: slint-gate init --namespace my-operator-system
# Generated at: 2026-04-30T12:00:00Z
# Metrics source: my-operator-controller-manager-metrics-service

schema_version: "slint.policy.v1"
thresholds:
  - name: "reconcile_total_delta_min"
    metric: "reconcile_total_delta"
    operator: ">="
    value: 1                          # 현재 관측값 기반 초기값
  - name: "workqueue_depth_end_max"
    metric: "workqueue_depth_end"
    operator: "<="
    value: 0                          # 현재 관측값 기반 초기값
regression:
  enabled: false                      # 첫 실행: baseline 없으므로 비활성
  tolerance_percent: 10
reliability:
  required: false
  min_level: "partial"
fail_on:
  - "threshold_miss"
```

**핵심**: 사용자가 임계값을 추측하지 않아도 된다. 실제 메트릭 값에서 출발한다.

---

### 4-2. 진단형 오류 메시지 [높음]

**목적**: 실패했을 때 무엇을 고쳐야 하는지 즉시 알 수 있게 한다.

**현재**:
```json
{
  "gate_result": "NO_GRADE",
  "reasons": ["MEASUREMENT_INPUT_MISSING"]
}
```

**개선 후**:
```
Gate Result: NO_GRADE

진단 결과:
  [MEASUREMENT_INPUT_MISSING]
  sli-summary.json 파일이 없거나 비어 있습니다.

  가능한 원인:
  1. E2E 테스트 실행 전에 slint-gate를 실행했습니다.
     → sess.End(ctx) 이후에 slint-gate를 실행하세요.

  2. sess.End(ctx)가 오류로 중단되었습니다.
     → E2E 테스트 로그에서 "fetch failed" 메시지를 확인하세요.
     → RBAC: ServiceAccount에 pods/create 권한이 있는지 확인하세요.
        kubectl auth can-i create pods --as=system:serviceaccount:<ns>:<sa> -n <ns>

  3. ArtifactsDir 경로가 잘못되었습니다.
     → SessionConfig.ArtifactsDir 값을 확인하세요.
```

RBAC 오류에 대해서는 `kubectl auth can-i` 명령을 직접 제시한다.

### 4-3. 기본 curl 이미지 제공 또는 InClusterFetcher 대안 [중간]

**목적**: CurlImage 설정 없이도 동작하게 한다.

**방향 A — 공개 기본 이미지**:
```go
// CurlImage 미지정 시 공개 이미지를 기본값으로 사용
CurlImage: "", // → docker.io/curlimages/curl:8.11.0 자동 적용
```

에어갭 환경을 위해 오버라이드는 유지한다.

**방향 B — InClusterFetcher (Go native)**:

curl-pod를 띄우는 대신, Go HTTP 클라이언트로 직접 metrics 엔드포인트를 호출하는 fetcher를 추가한다.

```go
// 현재 (curl-pod 방식)
fetcher := curlpod.NewFetcher(...)

// 추가 예정 (Go native 방식)
fetcher := incluster.NewFetcher(incluster.FetcherConfig{
    ServiceName: "my-operator-metrics-service",
    Namespace:   "my-operator-system",
    Port:        8443,
    TLSInsecureSkipVerify: true,
})
```

curl-pod 의존 없이 동작하면 RBAC 요구사항도 단순해진다.

### 4-4. kind 기반 Self-contained Quickstart [중간]

**목적**: 자기 클러스터 없이, 자기 오퍼레이터 없이, 5분 안에 첫 PASS를 본다.

```sh
git clone github.com/HeaInSeo/kube-slint
cd kube-slint
make quickstart
```

`make quickstart`가 하는 일:
1. kind 클러스터 생성
2. 샘플 오퍼레이터(hello-operator) 배포
3. E2E 테스트 실행 → `artifacts/sli-summary.json` 생성
4. `slint-gate` 실행 → `slint-gate-summary.json` 생성
5. 결과 출력:

```
============================
  kube-slint Quickstart
============================
Gate Result : PASS
Evaluation  : evaluated
Checks      : 4 passed, 0 failed

첫 PASS를 받았습니다!
다음 단계: docs/notes/onboarding-ux-improvement-2026-04-30.md 참고
============================
```

`test/consumer-onboarding/` 에 있는 자산을 활용하되, 외부 사용자가 혼자 실행할 수 있는 수준으로 완성한다.

### 4-5. GitHub Composite Action 제공 [낮음]

**목적**: GitHub Actions 사용자에게 copy-paste 없이 kube-slint를 적용하게 한다.

```yaml
# 사용 예시 (소비자 repo의 workflow)
- uses: HeaInSeo/kube-slint/.github/actions/slint-gate@v1
  with:
    measurement-summary: artifacts/sli-summary.json
    policy: .slint/policy.yaml
    baseline: docs/baselines/sli-summary.json
```

slint-gate 빌드, 실행, step summary 렌더링, gate_result 확인을 모두 액션 내부에서 처리한다.

## 5. 개발 순서 (Phase 계획)

### Phase 1 — 온보딩 마찰 제거 (즉시 착수 가능)

| 작업 | 산출물 | 예상 규모 |
|---|---|---|
| `slint-gate init` 구현 | `cmd/slint-gate` 서브커맨드 추가 | 중간 |
| 진단형 오류 메시지 | `internal/gate` 진단 레이어 추가 | 소 |
| 기본 curl 이미지 기본값 설정 | `SessionConfig` 기본값 추가 | 소 |

**완료 기준**: 외부 팀이 빈 namespace에서 `slint-gate init`을 실행하면 동작하는 `policy.yaml`과 `SessionConfig` 스니펫을 얻는다.

---

### Phase 2 — 클러스터 의존성 완화

| 작업 | 산출물 | 예상 규모 |
|---|---|---|
| `InClusterFetcher` 구현 | `pkg/slo/fetch/incluster` 패키지 | 중간 |
| kind quickstart | `Makefile` + `hack/quickstart.sh` | 중간 |
| RBAC 매니페스트 자동 생성 | `slint-gate init --emit-rbac` 플래그 | 소 |

**완료 기준**: `make quickstart`가 클린 환경에서 실패 없이 첫 PASS를 출력한다.

---

### Phase 3 — 생태계 연결

| 작업 | 산출물 | 예상 규모 |
|---|---|---|
| GitHub Composite Action | `.github/actions/slint-gate/action.yml` | 소 |
| kubebuilder plugin 검토 | 설계 문서 | 조사 |
| 공개 사용 사례 문서화 | `docs/case-studies/` | 소 |

## 6. 설계 제약

- **비침투 원칙 유지**: `slint-gate init`은 소비자 코드를 수정하지 않는다. 파일 생성과 스니펫 출력만 한다.
- **기존 계약 불변**: `sli-summary.json`, `slint-gate-summary.json` 스키마는 바꾸지 않는다.
- **선택적 기능**: InClusterFetcher는 curl-pod 방식을 대체하지 않는다. 둘 다 지원하고 사용자가 선택한다.
- **오버엔지니어링 방지**: Phase 1은 단순하게 유지한다. kubebuilder 자동 감지 같은 복잡한 휴리스틱은 Phase 2 이후에 검토한다.

## 7. 핵심 지표

이 개선이 성공했는지 판단하는 기준:

| 지표 | 현재 | 목표 |
|---|---|---|
| 첫 PASS까지 소요 시간 (외부 팀) | ~수 시간 | 30분 이내 |
| 온보딩 중 막히는 단계 수 | 5개 | 1개 이하 |
| `slint-gate init` → 유효한 policy.yaml | 불가 | 가능 |
| 클러스터 없이 동작 확인 | 불가 | quickstart로 가능 |

## 8. 열린 질문 (검토 필요)

1. **`slint-gate init`의 클러스터 접근 방식**: kubeconfig를 직접 사용할지, 환경 변수(`KUBECONFIG`)를 따를지 결정 필요.
2. **InClusterFetcher 인증**: ServiceAccount 토큰 vs. kubeconfig 기반 인증 중 기본값 선택 필요.
3. **기본 curl 이미지 public 노출 여부**: 공개 이미지를 기본값으로 하드코딩하면 버전 관리 책임이 생김. 별도 정책 필요.
4. **quickstart 대상 OS**: kind가 Linux/macOS를 지원하지만, CI 환경(GitHub Actions) 우선으로 할지, 로컬 개발 환경 우선으로 할지.
5. **Phase 1 착수 시점**: 현재 진행 중인 다른 작업과의 우선순위 조율 필요.
