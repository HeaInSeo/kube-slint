## Priority 0: Security and Gate Safety

### Ticket: Default-deny external ServiceURLFormat

Background:

kube-slint may scrape metrics through a curl pod and ServiceAccount token.
If a configured ServiceURLFormat points to an external host, an Authorization
header or mounted token could be exposed outside the cluster trust boundary.

Required behavior:

- Default mode accepts only cluster-local metrics hosts.
- External hosts are rejected before measurement begins unless an explicit
  dangerous opt-in exists.
- Service and namespace interpolation values are validated before URL
  construction.

Rejected behavior:

- Sending Authorization headers to arbitrary external hosts.
- Accepting `%s.%s.evil.example.com` as a service-local URL.
- Accepting malformed service or namespace values.

Acceptance criteria:

- External host URL is rejected by default.
- `.svc` and `.svc.cluster.local` handling is explicitly documented.
- Invalid scheme, host, service name, or namespace name returns a config or
  security error before scraping.

Test cases:

- `https://evil.example.com/collect?svc=%s&ns=%s`
- `https://%s.%s.evil.com/metrics`
- `ftp://%s.%s.svc/metrics`
- `https://<service>.<namespace>.svc:8443/metrics`
- `https://<service>.<namespace>.svc.cluster.local:8443/metrics`

Docs to update:

- `docs/security/service-url-format.md`
- `docs/security/token-handling.md`
- `docs/test-matrix/bad-fixtures.md`

Security impact:

Prevents token exfiltration through unsafe metrics URL configuration.

### Ticket: ServiceAccount token handling contract

Background:

Post-RC hardening removed direct bearer token command exposure. The remaining
contract should make token handling and redaction expectations explicit.

Required behavior:

- Tokens must not appear in command args, logs, JSON summaries, or error
  messages.
- Authorization material must be redacted from debug output.
- Token use must be scoped to the namespace-local scrape path.

Rejected behavior:

- Literal bearer token in curl args.
- Raw Authorization header in logs.
- Token forwarding to non-cluster-local hosts.

Acceptance criteria:

- Redaction rules cover `Authorization`, `Bearer`, `token=`, `password=`,
  `passwd=`, and `secret=` shapes.
- Tests demonstrate that command and error output do not leak token material.

Test cases:

- Curl command construction with token-backed auth.
- Runner error containing an Authorization header.
- JSON/YAML/CLI-flag shaped secret redaction.

Docs to update:

- `docs/security/token-handling.md`
- `docs/security-defaults.md`

Security impact:

Reduces CI log and Kubernetes command surface secret leakage.

### Ticket: Namespace-scoped RBAC by default

Background:

kube-slint should not require cluster-wide permissions for normal measurement
flows.

Required behavior:

- Default generated RBAC uses ServiceAccount, Role, and RoleBinding in the
  target namespace.
- ClusterRoleBinding requires explicit dangerous opt-in or a separate advanced
  path.

Rejected behavior:

- ClusterRoleBinding generated by the default path.
- Docs implying cluster-wide RBAC is the normal setup.

Acceptance criteria:

- Default RBAC examples are namespace-scoped.
- Cluster-wide RBAC, if supported, is marked as dangerous/advanced.

Test cases:

- RBAC manifest generation default path.
- Documentation scan for default ClusterRoleBinding wording.

Docs to update:

- `docs/security/rbac-model.md`
- `README.md`
- `docs/security-defaults.md`

Security impact:

Limits blast radius of kube-slint credentials and generated resources.

### Ticket: Invalid summary and policy must not PASS

Background:

As a CI gate, kube-slint must not silently treat malformed input as a passing
run.

Required behavior:

- Missing or unsupported summary schema version is rejected.
- Duplicate result IDs are rejected.
- Unknown result statuses are rejected.
- Missing or unsupported policy version is rejected.
- Duplicate threshold names are rejected.
- Unknown operators are rejected.
- Non-finite numeric values are classified by documented policy.

Rejected behavior:

- Unknown enum values ignored.
- Malformed policy producing PASS.
- Duplicate IDs causing last-write-wins behavior without an error.

Acceptance criteria:

- Invalid summary or policy produces invalid input status and non-PASS gate
  result.
- `FAIL`, `NO_GRADE`, and `WARN` priority is documented and tested.

Test cases:

- `missing-schema-version.json`
- `wrong-schema-version.json`
- `duplicate-result-id.json`
- `unknown-result-status.json`
- `missing-policy-version.yaml`
- `duplicate-threshold-name.yaml`
- `unknown-operator.yaml`
- `nan-threshold-value.yaml`

Docs to update:

- `docs/spec/summary-schema.md`
- `docs/spec/policy-schema.md`
- `docs/spec/gate-result-semantics.md`
- `docs/test-matrix/bad-fixtures.md`

Security impact:

Prevents malformed or adversarial gate input from being interpreted as safe.

## Priority 1: Test Matrix and Static Guardrails

### Ticket: Bad fixture matrix

Background:

Negative fixtures should define the expected behavior before implementation
fills in concrete files.

Required behavior:

- Summary, policy, and security bad fixtures each have a filename, input shape,
  expected result, and owner.

Rejected behavior:

- Fixture names without expected behavior.
- Security fixtures that are not tied to a guardrail decision.

Acceptance criteria:

- Matrix covers schema, duplicate, enum, NaN/Inf, external URL, insecure TLS,
  cluster-wide RBAC, privileged pod, hostPath, and unsafe cleanup cases.

Test cases:

- Listed in `docs/test-matrix/bad-fixtures.md`.

Docs to update:

- `docs/test-matrix/bad-fixtures.md`
- `testdata-plan/bad-fixtures/README.md`

Security impact:

Turns risky input classes into explicit regression tests.

### Ticket: Semgrep/custom guardrail rule plan

Background:

Generic Go linters do not know kube-slint-specific dangerous patterns.

Required behavior:

- Define rule IDs, intent, positive examples, and negative examples.
- Keep rules as planned guardrails until CI adoption is explicitly approved.

Rejected behavior:

- Enabling noisy CI-blocking rules without examples.
- Rules that conflate implementation style with security behavior.

Acceptance criteria:

- Rule plan includes:
  - `kube-slint-no-direct-service-url-format`
  - `kube-slint-no-bearer-token-in-curl-args`
  - `kube-slint-no-insecure-skip-verify`
  - `kube-slint-no-clusterrolebinding-default`
  - `kube-slint-no-stat-before-write`
  - `kube-slint-no-unsafe-cleanup`

Test cases:

- One positive and one negative sample for each rule.

Docs to update:

- `.semgrep/rules-plan.md`
- `.semgrep/kube-slint.yml` only after implementation approval.

Security impact:

Provides automated checks for patterns that are easy to miss in review.

## Priority 2: External User Readiness

### Ticket: kind E2E acceptance matrix

Background:

External users need confidence that kube-slint behavior is tested against real
Kubernetes paths, including negative scenarios.

Required behavior:

- Define kind E2E scenarios for RBAC, missing metrics service, endpoint 500,
  malformed metrics, external ServiceURLFormat, cleanup, parallel sessions,
  first-run baseline, regression, invalid policy, and invalid summary.

Rejected behavior:

- Happy-path-only E2E planning.
- Treating measurement failure as app correctness failure.

Acceptance criteria:

- Each scenario has setup, expected result, expected exit behavior, and
  artifact expectations.

Test cases:

- E2E-1 through E2E-12 in `docs/test-matrix/kind-e2e.md`.

Docs to update:

- `docs/test-matrix/kind-e2e.md`
- `docs/test-matrix/e2e-acceptance.md`

Security impact:

Validates guardrail behavior in realistic Kubernetes execution paths.

### Ticket: Release and GitHub Action policy

Background:

External CI users should not depend on `go run` for stable gate execution.

Required behavior:

- Define versioned binary, checksum, GitHub Action, semantic versioning, and
  changelog expectations.

Rejected behavior:

- Treating source checkout plus `go run` as the long-term stable user path.

Acceptance criteria:

- GitHub Action target usage is documented.
- Release artifact expectations are listed.
- Breaking change policy is drafted.

Test cases:

- Release checklist review.
- GitHub Action input/output contract review.

Docs to update:

- `docs/release/release-policy.md`
- `docs/release/versioning.md`
- `docs/integrations/github-action.md`

Security impact:

Improves reproducibility and reduces CI supply-chain ambiguity.

## Priority 3: Long-Term Maintainability

### Ticket: Schema compatibility policy

Background:

Summary and policy files are external contracts and need compatibility rules.

Required behavior:

- Define schema versioning, unknown field handling, deprecated fields,
  migration guidance, and old/new reader behavior.

Rejected behavior:

- Breaking schema changes without a major schema version.
- Ignoring unknown statuses or operators.

Acceptance criteria:

- Compatibility policy distinguishes metadata append-compatibility from
  semantic enum changes.

Test cases:

- Old summary read by new gate.
- New summary read by old gate.
- Unknown metadata field.
- Unknown result status.

Docs to update:

- `docs/spec/schema-compatibility.md`
- `docs/spec/migration-policy.md`

Security impact:

Reduces ambiguity in automated gate consumers and downstream tooling.

### Ticket: MCP read-only boundary

Background:

MCP can help explain kube-slint output, but it must not replace core gate
logic or mutate cluster/CI state in the initial integration.

Required behavior:

- Define read-only tools and forbidden write-capable tools.
- Define input and output contracts for explanation tools.
- Define resources MCP may read and must not modify.

Rejected behavior:

- MCP tools that apply RBAC, modify clusters, delete resources, or rewrite
  policy files.
- MCP positioned as kube-slint core functionality.

Acceptance criteria:

- Allowed tools include `explain_summary`, `explain_gate_result`,
  `analyze_policy`, `analyze_dataplane_manifest`, and constrained
  `suggest_fix`.
- Forbidden tools include `apply_rbac`, `modify_cluster`,
  `delete_resources`, `rewrite_policy`, and `run_privileged_check`.

Test cases:

- Read-only explain output examples.
- Rejection examples for write-capable operations.

Docs to update:

- `docs/integrations/mcp-readonly.md`
- `docs/security/mcp-boundary.md`

Security impact:

Prevents AI integration from expanding kube-slint's trust boundary too early.
